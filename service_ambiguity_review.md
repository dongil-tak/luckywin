## 1. 🚨 법적/신뢰도 리스크가 매우 높은 영역

### ① 명예의 전당 내 가짜 적중 데이터 (`LuckyHistory.tsx`)
*   **현황**: '명예의 전당' 화면에서 서비스가 1등 번호를 예측하여 적중시켰다는 배너(`🏆 서비스 예측 적중`)와 함께 당첨자 수(`윈웨이 서비스 배출 1명`)가 노출되고 있습니다. 현재 코드는 `index % 3 === 0`과 같은 **가상의 조건(Mocking)으로 적중 여부를 조작**하여 보여주고 있습니다.
*   **문제점**: 실제 백엔드 연동 없이 가상으로 1등 당첨을 다수 배출한 것처럼 구성하는 것은 표시광고법상 **'허위·기만적 광고'**에 해당하여 출시 즉시 법적 제재를 받을 수 있습니다. 사용자들 또한 매주 동일한 가짜 적중 정보가 반복되는 것을 보면 서비스 전체의 신뢰도를 의심하게 됩니다.
*   **개개 가이드**: 실제 사용자가 앱 내에서 생성하여 당첨된 실제 이력이 쌓이기 전까지는 이 배지를 **비활성화(Won't)** 처리하거나, 단순 '역대 공식 당첨 번호 조회' 기능으로만 제공하십시오.

### ② 비현실적인 서비스 적중률 수치 (`Management.tsx`)
*   **현황**: 금주의 당첨현황 화면에서 **'우리 서비스의 적중률: 83%'**라는 수치가 도넛 차트로 강조되어 있습니다. 설명에는 *"모든 유저 생성 조합 중 1등 번호 포함 비율"*이라고 되어 있습니다.
*   **문제점**: 로또 1등 당첨 확률(약 814만분의 1)을 고려할 때 '83%'라는 수치는 수학적으로 불가능하며, 사용자가 오해하기 매우 쉽습니다. 만약 6개 번호 중 1~2개 맞춘 것을 포함한 수치라면 **'적중률 83%'가 아닌 '번호 일치율(1개 이상 일치)' 등으로 정밀하게 재정의**해야 하며, 그렇지 않으면 사기성 번호 생성기라는 인상을 줍니다.
*   **개선 가이드**: 오해의 소지가 있는 83% 수치 디자인을 삭제하고, **"이번 주 당첨 번호 중 3개 이상 일치한 유저 조합 수: OO건"**과 같이 팩트에 기반한 직관적인 숫자로 교체하십시오.

## 2. ❓ 개념 및 용어가 혼재되어 혼란을 주는 영역

### ① '사주/운세(Fortune)' 폴더명과 '로또 분석' 콘텐츠의 불일치
*   **현황**: 소스코드 구조상 핵심 라우트가 `/fortune`으로 되어 있고, 화면 타이틀도 '나의 행운 관리' 등으로 되어 있으나, 실제 제공되는 기능은 100% **'로또(Lotto) 번호 생성 및 당첨 분석'**입니다. 사주조회나 탄생화, 타로 등 '행운/운세'와 관련된 서비스 콘텐츠는 전혀 구현되어 있지 않습니다.
*   **문제점**: 사용자는 메뉴나 경로명에서 '운세/사주 기반 로또 추천'을 기대했으나 실제로는 단순 통계/난수 기반 생성기만 보게 되어 브랜드 정체성(Identity)에 혼란을 느낍니다.
*   **개선 가이드**:
    *   사주/운세 데이터를 수집하여 추천하는 기능을 실제로 추가할 것인지, 아니라면 서비스 브랜딩에서 `Fortune` 관련 키워드를 제거하고 `Lotto/Lucky`로 일원화할 것인지 결정해야 합니다.
    *   `/fortune` 관련 파일 경로 및 라우트명을 서비스 아이덴티티에 맞게 `/result` 또는 `/winning-status` 등으로 리팩토링하는 것을 권장합니다.

### ② 분석(Analysis)과 선택(Selection)의 용어 혼선 (`AIAnalysis.tsx` / `BottomNav.tsx`)
*   **현황**: 하단 탭바에는 **'번호 직접선택'**으로 되어 있으나, AI 자동 화면의 세그먼트 컨트롤에는 **'수동 분석'**이라는 이름으로 동일한 화면(`/dashboard`) 링크가 매핑되어 있습니다.
*   **문제점**: 사용자가 번호를 손으로 직접 고르는 행위는 '직접 선택'이지 '분석'이 아닙니다.
*   **개선 가이드**: 용어를 **'수동 선택'**과 **'AI 자동 추천'**으로 직관적이고 일관되게 통일해야 혼란을 줄일 수 있습니다.

## 3. ⚙️ 사용성 관점에서 불필요하거나 모호한 영역

### ① 사용자 화면에 노출된 개발 로드맵 및 API 계획 (`LottoStore.tsx`)
*   **현황**: 로또 복권 판매소 화면 하단에 **'API 연동 계획(공공데이터 포털 연동 예정, 지도 연동 예정 등)'**이 체크리스트 카드로 최종 사용자에게 노출되고 있습니다.
*   **문제점**: 개발 예정 스펙이나 API 연동 상태를 최종 사용자 화면에 상세히 보여주는 것은 서비스가 미완성 상태임을 자인하는 꼴이며, 사용자에게 불필요한 정보입니다.
*   **개선 가이드**: 해당 정보를 화면에서 삭제 처리하고, 실제 연동 완료 전까지는 해당 섹션을 제거하거나 '서비스 준비 중' 팝업 하나로 대체해야 합니다.

### ② '나의 수동 조합함'의 5세트 제한 및 저장 플로우의 모호성 (`Dashboard.tsx`)
*   **현황**: 번호 6개를 골라 '세트 저장'을 누르면 즉시 기기에 영구 저장되는 것이 아니라, 화면 중간 '나의 수동 조합함'에 임시 적재됩니다. 그 후 하단의 '번호 저장하기'를 한 번 더 눌러야 `localStorage`에 최종 저장됩니다.
*   **문제점**: 사용자는 '세트 저장' 버튼을 누른 순간 이미 저장이 완료되었다고 인지할 확률이 높습니다. 이 상태에서 페이지를 이탈하면 임시 조합이 모두 날아가는 UX 참사가 발생합니다.
*   **개선 가이드**: '세트 추가' 시 즉시 로컬 스토리지에 저장하고 조합함을 갱신하도록 플로우를 단순화하여 이중 저장 단계로 인한 모호함을 제거해야 합니다.
